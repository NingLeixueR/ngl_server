# ngl_server 项目评级报告

## 评估范围

- 评估日期：`2026_04_23`
- 评估对象：`ngl_server`
- 评估方式：基于仓库静态检查，不包含编译、运行、压测和线上环境验证
- 评估目标：从架构、工程化、测试、可维护性和持续演进能力几个维度，给出当前项目成熟度评级、优缺点总结，以及下一阶段优化路线

## 总结论

当前项目整体处于“架构思路成熟、工程化基础较强、具备持续演进能力，但在自动化质量门禁、模块边界和配套平台成熟度上仍有明显提升空间”的阶段。

- 综合评级：`A-`
- 综合评分：`7.8 / 10`

这个项目已经不是“演示级”代码库，而是一个具备明确技术方向、实际可维护意图和跨平台构建能力的服务端框架。它的核心优势在于 Actor 架构、较完整的网络协议栈、较系统的文档组织，以及已经成型的 CMake + Conan + GoogleTest 工程链路。短板主要集中在自动测试门禁仍偏弱、构建耦合偏高、部分辅助模块成熟度不均衡，以及部分开发体验问题还没有完全收口。

## 客观观察

从当前仓库快照看，可以得到一些比较明确的工程信号：

- `public/cpp` 与 `server` 下可见源码文件约 `404` 个，说明项目已达到中大型代码库体量
- `server/tests` 下共有 `13` 个 `*_gtest.cpp` 测试文件，静态统计到约 `149` 个 `TEST/TEST_F`
- CI 工作流包含 `Windows` 构建、`Linux` 构建、手动测试流程和 `clang-format` 检查
- 顶层构建已经使用 `C++20`、`Conan 2`、`BehaviorTree.CPP`、`Boost`、`Protobuf`、`libcurl`、`MySQL`、`PostgreSQL`、`Redis` 等较完整的服务端依赖集合
- 文档目录 `doc/` 已经具备目录说明、模块说明、架构说明和优化方案沉淀，不再是单纯的 README 驱动项目

这些信号说明：项目不是“代码很多但工程化很弱”的类型，而是“明确开始把可维护性、跨平台构建、测试和文档纳入长期治理”的类型。

## 主要优点

### 1. 核心架构方向正确

Actor 模型是这个项目最突出的架构资产。调度层、线程池、消息队列和按层路由的设计都体现出较强的并发场景意识。尤其是对单例 Actor 与动态 Actor 的分层分桶策略，说明作者已经在主动规避热点集中和简单取模导致的负载偏斜问题。

这类设计不是“写出来能跑”的层面，而是已经开始考虑长期扩展性和运行时行为特征。

### 2. 网络栈与协议能力比较完整

项目同时覆盖了 `TCP`、`WebSocket/WSS`、`KCP`，并且配套了协议封包、配置系统、客户端实现和机器人路径，这意味着它不是单点实验，而是朝“完整服务端基础设施”方向组织的。

对于游戏服务端框架而言，这一点很关键。很多项目只会把网络接入做成一个单点模块，而这里已经形成了协议、连接、配置、客户端验证和测试联动。

### 3. 工程化基础明显强于普通个人项目

当前仓库已经具备以下成熟信号：

- Conan 管理第三方依赖
- CMake 统一构建入口
- 可开关的强告警、Sanitizer、Unity Build
- Windows / Linux 双平台 CI
- GoogleTest 测试组织

这意味着项目已经从“单机开发脚本式工程”升级到“可复用、可迁移、可协作”的工程体系。

### 4. 文档沉淀意识很强

`doc/` 目录已经不是零散笔记，而是按模块、目录、架构、调用链路和专项优化来组织。这个特征非常重要，因为长期项目最终比拼的不是“写功能的速度”，而是“后来的人能不能接着维护”。

文档在这个项目里已经开始承担知识传递和维护入口的作用，这是非常积极的信号。

### 5. 测试覆盖面较广

当前测试不是只停留在工具函数层面，而是已经覆盖到：

- AI/行为树
- runtime
- actor 管理
- 网络层
- 序列化
- 配置与 XML
- smoke
- 性能基准

这说明项目已经开始形成“有核心回归保护网”的状态。

## 主要缺点

### 1. 主 CI 缺少自动测试门禁

这是当前最明显的工程短板。

现有 Windows / Linux 主工作流会构建并上传产物，但测试执行主要放在 `manual-test.yml` 中。这会导致一个现实问题：代码可以在 PR 或 push 阶段编译成功，但没有被自动回归测试真正阻断。

换句话说，项目当前更接近“会构建”而不是“会自动证明正确”。

这个问题优先级很高，因为它直接影响后续大规模重构和多人协作的信心。

### 2. 构建粒度偏粗，模块边界仍然不够轻

`nglcore` 对 `public/cpp` 源文件的聚合方式比较重，配合 `GLOB_RECURSE` 和 Unity Build，虽然在短期内能降低维护 source list 的成本，但长期会带来这些问题：

- 单个模块改动容易触发较大范围重编译
- 目标边界不清晰，耦合关系难以量化
- 更细粒度的测试、链接裁剪和复用会变得困难

这不是“不能用”，但会成为中后期演进的效率瓶颈。

### 3. 质量检查覆盖范围还不够全面

当前 `clang-format` 工作流只覆盖 `public/cpp`，而 `server/`、`client/`、`tools/` 等目录并没有被同等约束。随着项目继续扩张，这会让风格一致性和代码卫生出现分层落差。

另外，仓库里还没有看到 `clang-tidy`、覆盖率采集或更强的静态分析流水线配置，这意味着质量治理仍偏基础层。

### 4. 工具链和配套模块存在平台成熟度不均衡

`server/tools` 仅在本地 Windows 构建路径中被启用，这说明工具模块还没有真正纳入统一跨平台工程视角。  
`client` 模块已经具备一定完整性，但 README 明确说明目前并未安装 `typescript` / `@types/node`。这意味着它更像“可用的工程资产”，而不是“被强约束验证的正式子产品”。

`admin` 目录内容不少，但文档非常薄弱，目前更像历史积累的运维后台资源，而不是一个被系统维护的现代化子系统。

### 5. 仓库卫生还有细节问题

例如顶层 `.gitignore` 中直接忽略 `client` 目录，这对已跟踪文件影响有限，但对后续新增文件和维护者心智都会造成混淆。类似这种问题不会立刻导致故障，却会长期侵蚀仓库可预测性。

## 适合的发展定位

如果从产品和技术方向上看，这个项目最适合继续演进成：

- 一个偏“游戏服务端基础框架”的长期仓库
- 一个可承载业务 Actor、协议与运营工具的统一平台
- 一个兼顾运行时、工具链、测试和客户端联调的完整基础设施项目

它不应该再停留在“继续堆功能”的阶段，而应逐步进入“控制复杂度、增强自动化、提炼平台层”的阶段。

## 后续优化方向

下面的优化路线按优先级分为三个阶段。

### 阶段一：补齐质量门禁

目标：让项目从“能编译”提升到“改动默认会被自动验证”。

优先动作：

1. 将 `ngl_test_fast` 纳入 Windows / Linux 主构建工作流，PR 自动执行
2. 保留手动测试流程，但把它定位为“全量/性能/产物验证”，而不是主验证入口
3. 为 Linux 增加 `Sanitizer` 夜间任务或手动触发任务
4. 不再忽略 `.github/**` 的 CI 触发路径，让工作流修改本身也能被验证

预期收益：

- 明显降低重构时的不确定性
- 提高多人协作时的代码信任度
- 为后续更大规模架构调整建立保护网

### 阶段二：梳理构建边界与模块拆分

目标：降低构建耦合，提高增量开发效率。

优先动作：

1. 将 `nglcore` 内部逐步拆分为更清晰的静态库边界，例如：
   - `actor_core`
   - `net_core`
   - `tools_core`
   - `config_core`
2. 对关键目录逐步减少 `GLOB_RECURSE` 的使用，改为显式 source list 或模块级 list 文件
3. 评估哪些工具模块可以直接复用 `nglcore`，哪些必须保留独立编译边界
4. 把“编译成本最高的模块”作为首批拆分对象，而不是追求一次性重构全部

预期收益：

- 减少大范围重编译
- 让模块依赖关系更透明
- 让测试、静态分析和链接优化更容易落地

### 阶段三：提升周边子系统成熟度

目标：让仓库从“核心 C++ 服务端强”升级为“整个平台协同强”。

优先动作：

1. 为 `client` 增加正式的类型检查和最小自动化测试
2. 为 `admin` 补充模块说明、部署方式、安全边界和维护入口文档
3. 把更多目录纳入统一格式化和静态检查治理
4. 逐步建立版本化工程指标，例如：
   - 构建时长
   - 测试时长
   - 测试失败率
   - 每次改动影响文件数
   - 回归缺陷数

预期收益：

- 提升非核心模块的可维护性
- 让整个仓库的成熟度更加均衡
- 为后续形成真正的平台型仓库打基础

## 建议的近期优先级排序

如果只做最关键的五件事，我建议顺序如下：

1. 主 CI 自动执行快速测试
2. 增加 Linux Sanitizer 验证
3. 调整 `.github` 工作流触发策略
4. 拆分 `nglcore` 的构建边界
5. 为 `client` 和 `admin` 补齐基本工程规范

## 最终评价

`ngl_server` 当前已经具备比较明显的“长期项目”特征：

- 不是临时实验代码
- 不是单一功能原型
- 已经具备平台雏形

它当前最大的任务，不是继续无节制扩展功能，而是把已有能力进一步“工程化、产品化、可验证化”。

如果后续两到三个阶段的优化能持续落地，这个项目有机会从现在的 `A- / 7.8`，提升到 `A / 8.4+`，并逐步进入“高质量基础框架项目”的稳定区间。
